Trusta AST Fine-tune Service User Manual

This document explains in detail the architecture, API workflow, dataset format, and complete operating guide of the fine-tuning service. It is intended for team training and onboarding.

Supported methods: LoRA / QLoRA / Full Parameter Fine-tuning. The model is automatically converted to GGUF Q4_K_M format after training completes.

A Traditional Chinese version of this manual is available: finetune_manual.html.

1. Architecture Overview

The Fine-tune Service uses the same multi-process isolation architecture as the Inference Service. Training jobs run in a dedicated Worker Process, ensuring that an OOM or a crash does not affect the main FastAPI service.

graph TD Client[Client / Frontend] -- HTTP API --> API[FastAPI Server (service/app.py)] subgraph MainProcess [Main Process] API -- calls --> TM[TrainingManager (Singleton)] TM -- delegates --> TPM[TrainingProcessManager] TPM -- stores --> Redis[(Redis / Memory)] MR[ModelRegistry] -- registers finetuned model --> Registry[(models_registry.json)] end subgraph WorkerProcess [Isolated Training Worker Process] TPM -- Queue IPC --> Worker[_training_worker_process] Worker -- 1. Strategy --> Strategy[StrategyFactory] Worker -- 2. Load --> Tokenizer[AutoTokenizer] Worker -- 2. Load --> Dataset[Dataset JSONL] Worker -- 3. Load --> Model[AutoModelForCausalLM + PEFT] Worker -- 4. Train --> Trainer[SFTTrainer / CustomTrainer] Trainer -- 5. Save --> OutputDir[output_dir/] OutputDir -- 6. Convert --> GGUF[GGUF Q4_K_M file] OutputDir -- 7. Register --> MR end Worker -- progress / status --> TPM
Important: In every case (completion / stop / error), the training process exits the Worker Process, ensuring that VRAM is fully released. A clean Worker is started again for the next training run.

2. Training Methods Comparison

LoRA

Low-Rank Adaptation

  • Trains only low-rank matrices; very few parameters
  • Low GPU requirements
  • Fast training speed
  • Output: adapter weight file

QLoRA

Quantized LoRA (4-bit)

  • Base model is loaded with 4-bit quantization
  • Even lower GPU requirements (suitable for consumer-grade GPUs)
  • Slight accuracy loss
  • Output: adapter weight file

Full Parameter

Full Parameter Fine-tuning

  • Trains all parameters
  • Requires large amounts of GPU VRAM
  • Best results
  • Usually paired with DeepSpeed ZeRO

3. Key APIs and Workflow

All training-related APIs are prefixed with /training/ and defined in service/app.py.

1. Start Training

POST /training/start

Behavior: Synchronous validation + asynchronous training.

Pass a complete TrainingConfig. Once validation passes, a session_id is returned immediately. The actual training runs in the background in the Worker Process.

Note: If output_dir already contains files, the API returns an error (it must be cleared manually).

2. Query Training Status

GET /training/status

Behavior: Synchronous, used for polling.

Returns a TrainingStatus containing is_training, progress (0–1), current_step, total_steps, loss, current_epoch, status, and error.

3. Stop Training

POST /training/stop

Behavior: Forcibly terminates the Worker Process (terminate + kill).

Does not wait for the Worker to exit gracefully; it terminates the process directly and resets the state. Checkpoints that have already been saved are not affected.

4. Training History

GET /training/status/{session_id}/history

Returns the detailed loss / learning_rate / epoch records for that training session.

The data is stored in Redis (if available) or in memory, indexed by session_id.

5. System Resource History

GET /system/resource/{session_id}/history

Returns the CPU / GPU / Disk usage history during training, sampled once per logging_steps.

6. Error Details

GET /training/error_details

Retrieves the complete training error traceback, including the is_oom flag. The format is consistent with /inference/error_details.

7. Force GPU Cleanup

POST /training/force_cleanup_gpu

Forcibly terminates the Training Worker Process and resets the state. Used when VRAM has not been released after an OOM (does not affect the Inference Worker).

Complete Training Workflow Example (Python)

import requests, time

BASE = "http://localhost:8000"

# Step 1: Start training
config = {
    "model_name": "Qwen/Qwen3-4B",
    "method": "lora",
    "dataset_path": "dataset/mydataset.jsonl",
    "output_dir": "./output/qwen3_lora_v1",
    "lora_r": 16,
    "lora_alpha": 32,
    "lora_dropout": 0.05,
    "num_train_epochs": 3,
    "per_device_train_batch_size": 1,
    "gradient_accumulation_steps": 8,
    "learning_rate": 2e-4,
    "max_seq_length": 2048,
    "prompt_field": "prompt",
    "completion_field": "completion",
    "use_sft_trainer": True
}

resp = requests.post(f"{BASE}/training/start", json=config)
session_id = resp.json()["result"]["session_id"]
print(f"Training started: session_id={session_id}")

# Step 2: Poll progress
while True:
    status = requests.get(f"{BASE}/training/status").json()
    if status["is_training"]:
        print(f"  Step {status['current_step']}/{status['total_steps']} "
              f"| Progress: {status['progress']:.1%} "
              f"| Loss: {status.get('loss', 'N/A'):.4f}"
              f"| Epoch: {status.get('current_epoch', 0):.2f}")
        time.sleep(10)
    else:
        print(f"Training ended: status={status['status']}")
        if status.get("error"):
            print(f"Error: {status['error']}")
        break

# Step 3: Retrieve the detailed loss curve
history = requests.get(f"{BASE}/training/status/{session_id}/history").json()
for log in history["logs"]:
    print(f"Step {log['step']}: loss={log['loss']:.4f}, lr={log.get('learning_rate')}")

4. Code Structure Explained

File Path Role and Function
service/app.py API entry point. Defines the /training/* routes and calls training_manager.
service/training_manager.py Singleton Facade. Exposes methods such as start_training, get_status, stop_training, and get_history to the API layer, and provides concurrency protection (Lock).
service/training/training_process.py Process control core. Includes:
  • TrainingProcessManager: Manages the Worker Process lifecycle, IPC Queue, status parsing, and reading history from Redis.
  • _training_worker_process(): The main loop that actually runs in the child process, receiving start/stop commands.
service/training/core/strategies.py Strategy Pattern.
  • StrategyFactory: Selects a strategy based on use_sft_trainer.
  • SFTStrategy: Uses the TRL SFTTrainer (recommended; supports completion-only loss).
  • CausalLMStrategy: Uses the native HF Trainer (manual tokenization).
service/training/sft_runner.py SFTTrainer initialization. Creates CustomSFTTrainer (overriding save_model for compatibility with DeepSpeed + LoRA) and SFTConfig.
service/training/core/model_loader.py Model and tokenizer loading. Applies the appropriate BitsAndBytesConfig and LoraConfig depending on the method (LoRA/QLoRA/Full).
service/training/core/dataset_loader.py Dataset loading. Reads data in JSON or JSONL format and converts it into a HuggingFace Dataset object.
service/training/core/model_saver.py Result saving. Saves the model weights and tokenizer, and calls model_registry to register the fine-tuned model as a finetuned type.
service/utils/conversion_manager.py GGUF conversion. After training completes, automatically calls the llama.cpp conversion script to generate a Q4_K_M quantized GGUF file.
service/configs/deepspeed/ DeepSpeed configuration templates. Provides four preset ZeRO-3 Offload combinations:
  • zero3_offload_cpu_cpu.json: Both Optimizer & Param offloaded to CPU
  • zero3_offload_cpu_disk.json: Optimizer→CPU, Param→Disk
  • zero3_offload_disk_cpu.json: Optimizer→Disk, Param→CPU
  • zero3_offload_disk_disk.json: Both offloaded to Disk (NVMe)

5. Training Process Details (Deep Dive)

Worker Process Main Flow

1
Receive the Start command
The Worker takes {"command": "start", "config": {...}, "session_id": "..."} from the Request Queue. It checks whether output_dir already contains files (if so, it reports an error to prevent overwriting).
2
Strategy selection
StrategyFactory.get_strategy(config):
  • use_sft_trainer=True (default) → SFTStrategy (uses the TRL SFTTrainer)
  • use_sft_trainer=False → CausalLMStrategy (uses the native HF Trainer + manual tokenization)
At the same time, it initializes TrainingArguments (including the DeepSpeed configuration, BF16, and Gradient Checkpointing).
3
Load resources (executed in sequence to save memory)
  1. Load the Tokenizer (AutoTokenizer.from_pretrained).
  2. Load the Dataset (JSONL → HuggingFace Dataset).
  3. If CausalLMStrategy is used, the dataset is pre-tokenized at this point (to avoid CPU waits during training).
  4. Load the Model (applying the quantization settings and PEFT LoraConfig based on the method).
4
Prepare the Trainer
strategy.prepare_trainer(model, tokenizer, dataset, training_args).
  • SFTStrategy: Creates CustomSFTTrainer, automatically handling prompt+completion loss masking (completion_only_loss=True).
  • CausalLMStrategy: Creates CustomTrainer, using DataCollatorForLanguageModeling.
A _ProgressCallback is added, which reports progress to the Status Queue and stores it in Redis every logging_steps.
5
Run training
trainer.train(). During training, every logging_steps:
  • Pushes {"status": "progress", "step": N, "loss": ..., "epoch": ...} to the Status Queue.
  • Samples CPU/GPU/Disk resource usage and stores it in Redis.
6
Save results
save_training_results(trainer, tokenizer, training_config) saves the adapter or the full model into output_dir, and automatically calls model_registry to register it as a finetuned model.
7
Clean up resources + GGUF conversion
It first runs _cleanup_training_resources() (deleting the Trainer/Model/Tokenizer objects, running GC, and clearing the CUDA cache) to ensure VRAM is released, and then calls _convert_training_output_to_q4_k_m() to perform the GGUF conversion.
8
Report completion and exit
Pushes {"status": "completed", "gguf_path": "...", "gguf_quantization": "Q4_K_M"} to the Status Queue. The Worker Process then exits, ensuring all resources are released.

Progress State Machine

stateDiagram-v2 [*] --> idle idle --> starting: start_training() starting --> initializing: Worker receives command initializing --> running: trainer.train() begins running --> completed: Training completes successfully + GGUF conversion running --> stopped: stop command received running --> error: OOM or Exception occurs completed --> idle: next training run stopped --> idle: next training run error --> idle: after force_cleanup_gpu

6. Dataset Format

Training data must be in JSON or JSONL format (in JSONL, each line is one JSON object). Two training modes are supported, selected via text_field or prompt_field/completion_field.

Mode 1: Single-field text mode (full-sequence loss)

Suitable for: language model pre-training or scenarios that require full-sequence training. Set text_field: "text" (or the corresponding field name).

// dataset.jsonl (one record per line)
{"text": "### Question:\nHow do I set the time zone on a Linux system?\n\n### Answer:\nUse the timedatectl command: sudo timedatectl set-timezone Asia/Taipei"}
{"text": "### Question:\nWhat is a Docker volume?\n\n### Answer:\nA Docker volume is a mechanism for persisting container data..."}
{"text": "This is a piece of text content that the model needs to learn..."}

Mode 2: Dual-field prompt + completion mode (completion-only loss)

Suitable for: instruction fine-tuning. The SFTTrainer automatically masks the loss of the prompt and computes the loss only for the completion part.

Set prompt_field: "prompt", completion_field: "completion".

// mydataset.jsonl (one record per line)
{"prompt": "You are a Linux system administrator.\nHow do I check system memory usage?", "completion": "You can check memory usage with the free -h command:\n```\nfree -h\n```\nHere, -h displays the output in a human-readable format..."}
{"prompt": "Explain what the GIL is in Python", "completion": "The GIL (Global Interpreter Lock) is a mechanism in CPython..."}
{"prompt": "How do I undo the last commit in Git?", "completion": "There are several ways:\n1. Keep changes: git reset --soft HEAD~1\n2. Discard changes: git reset --hard HEAD~1"}
Recommended choice: For instruction fine-tuning, we recommend using the dual-field mode + use_sft_trainer: true. The system will automatically enable completion_only_loss=True, so the model learns only the answer portion, making training more efficient.

Dataset Quality Recommendations

RecommendationDescription
Minimum data volume At least 500–1000 records is recommended; data quality matters more than quantity
Sequence length Keep it within max_seq_length; anything longer will be truncated. It is recommended that 95% of the data not exceed the configured length
Data cleaning Remove duplicate data, ensure correct UTF-8 encoding, and avoid empty prompt/completion fields
Format consistency The prompt format must be consistent within the same dataset (e.g., whether a system prompt is added)

7. Configuration and Parameters

Complete TrainingConfig Reference

{
  // === Required ===
  "model_name": "Qwen/Qwen3-4B",        // Label in the Model Registry or an HF model ID
  "method": "lora",                       // Training method: lora / qlora / full
  "dataset_path": "dataset/train.jsonl",  // Dataset path (relative or absolute)
  "output_dir": "./output/my_model_v1",   // Output directory (must be an empty directory)

  // === LoRA/QLoRA only ===
  "lora_r": 16,                           // LoRA rank (larger -> more parameters, better results but slower)
  "lora_alpha": 32,                       // LoRA scaling (usually = lora_r * 2)
  "lora_dropout": 0.05,                   // Dropout (prevents overfitting)
  "lora_target_modules": [               // Modules to train (empty/null = auto-detect linear layers; only falls back to this list if detection fails)
    "q_proj", "k_proj", "v_proj", "o_proj",
    "gate_proj", "up_proj", "down_proj"
  ],

  // === Training hyperparameters ===
  "num_train_epochs": 3,                 // Number of training epochs
  "per_device_train_batch_size": 1,      // Batch size per step (reduce to 1 if memory is insufficient)
  "gradient_accumulation_steps": 8,      // Gradient accumulation steps (effective batch = batch * accum)
  "learning_rate": 2e-4,                 // Learning rate (1e-4 ~ 3e-4 is common for LoRA/QLoRA)
  "warmup_steps": 100,                   // Learning rate warmup steps
  "max_seq_length": 2048,               // Maximum sequence length (longer is truncated)

  // === Dataset field configuration (choose one) ===
  // Mode 1 (full-sequence loss):
  "text_field": "text",
  // Mode 2 (completion-only loss, recommended for instruction fine-tuning):
  "prompt_field": "prompt",
  "completion_field": "completion",

  // === Save settings ===
  "logging_steps": 10,                   // Report loss every N steps
  "save_steps": 500,                     // Save a checkpoint every N steps
  "save_total_limit": 2,                 // Maximum number of checkpoints to keep
  "save_tokenizer": true,                // Whether to save the tokenizer

  // === Trainer selection ===
  "use_sft_trainer": true,               // Use the TRL SFTTrainer (recommended)
  "packing": false,                      // Whether to enable sequence packing
  "gradient_checkpointing": true,        // Gradient checkpointing (saves VRAM)

  // === DeepSpeed (optional) ===
  "use_deepspeed": false,
  "deepspeed_profile": "zero3_offload_cpu_cpu",  // Use a built-in profile
  // or
  "deepspeed_config": "./my_deepspeed.json",       // Use a custom path
  "offload_folder": "./deepspeed_offload"          // ZeRO offload directory
}

8. DeepSpeed Offload Configuration

When GPU VRAM is insufficient to hold the full model (typically for Full parameter training or LoRA on large models), you can enable DeepSpeed ZeRO-3 Offload to offload part of the data to CPU RAM or NVMe Disk.

Note: DeepSpeed significantly lengthens training time (CPU offload is 3-5x slower, Disk offload is 5-20x slower). We recommend trying QLoRA + a smaller batch size first, and only enabling DeepSpeed when necessary.

Four Built-in Profiles

Profile Name Optimizer Offload Param Offload Use Case
zero3_offload_cpu_cpu CPU RAM CPU RAM Plenty of CPU RAM (>64GB); most commonly used
zero3_offload_cpu_disk CPU RAM NVMe Disk Insufficient CPU RAM, but fast NVMe available
zero3_offload_disk_cpu NVMe Disk CPU RAM Special scenarios
zero3_offload_disk_disk NVMe Disk NVMe Disk Extremely low-memory environments; requires high-speed NVMe

Configuration Example with DeepSpeed Enabled

{
  "model_name": "Qwen/Qwen3-14B",
  "method": "lora",
  "dataset_path": "dataset/train.jsonl",
  "output_dir": "./output/qwen3_14b_lora",
  "use_deepspeed": true,
  "deepspeed_profile": "zero3_offload_cpu_cpu",  // Use a built-in profile
  "offload_folder": "./deepspeed_offload",         // If the profile uses nvme, this path overrides it
  "gradient_checkpointing": true,
  "lora_r": 16,
  "lora_alpha": 32,
  "num_train_epochs": 3
}
Special handling for DeepSpeed + LoRA: The default save_model behavior of ZeRO-3 is incompatible with LoRA adapters (it may cause OOM or a FileExists error). The system already overrides save_model via CustomSFTTrainer and CustomTrainer, using GatheredParameters to safely gather the adapter parameters before saving, so this issue is handled automatically.

9. Post-Training Workflow

Automated post-processing: After training completes successfully, the system automatically performs the following steps, with no manual action required:

Automatic GGUF Conversion

  1. Training completes + the model is saved to output_dir.
  2. Training resources are cleaned up (VRAM released).
  3. conversion_manager.convert_and_quantize() is called:
    • First, the HF-format model is converted to GGUF F16.
    • Then it is quantized to Q4_K_M format (smaller size, faster inference).
  4. The GGUF file is stored inside output_dir.
  5. /training/status returns gguf_path and gguf_quantization.

Automatic Registration in the Model Registry

After training completes, the fine-tuned model is automatically registered in models_registry.json as a finetuned type. It can be queried via the following API:

GET /config/models
// The response includes:
{
  "finetuned_models": [
    {
      "label": "my_model_v1",
      "output_dir": "./output/my_model_v1",
      "base_model": "Qwen/Qwen3-4B",
      "method": "lora",
      ...
    }
  ]
}

Running Inference with the Fine-tuned Model

The trained model can be loaded directly via the Inference API (specify model_path as the output_dir):

POST /inference/load_model
{
  "model_name": "my_model_v1",        // Registry label
  "model_path": "./output/my_model_v1", // Fine-tune output directory
  "engine": "transformers",
  "quantization": "none"              // The adapter is already low-precision; no extra quantization needed
}

Or use the automatically converted GGUF model (recommended for production):

POST /inference/load_model
{
  "model_name": "my_model_v1_gguf",
  "model_path": "./output/my_model_v1/my_model_v1-Q4_K_M.gguf",
  "engine": "llama_server",
  "n_gpu_layers": -1
}

10. Troubleshooting

❌ Error: Output directory already exists and contains files

Cause: output_dir already contains leftover files from a previous training run.

Solution:

  • Manually clear or delete output_dir.
  • Or use a new directory name (e.g., add a version number: my_model_v2).
The system will not automatically clear the output directory, to prevent accidentally overwriting previous training results.

❌ CUDA Out of Memory (OOM) during training

Symptom: Training crashes, and is_oom=true in /training/error_details.

Solutions (in order of priority):

  1. Call POST /training/force_cleanup_gpu to ensure VRAM is released.
  2. Switch to method: "qlora" (4-bit quantized base model).
  3. Lower per_device_train_batch_size to 1 and increase gradient_accumulation_steps.
  4. Lower max_seq_length (2048 → 1024 → 512).
  5. Enable DeepSpeed CPU Offload: use_deepspeed: true, deepspeed_profile: "zero3_offload_cpu_cpu".

❌ Training Error: RuntimeError / CUBLAS_STATUS_ALLOC_FAILED

Cause: A PEFT device allocation issue in a multi-GPU environment.

Solution: The Worker Process already sets CUDA_VISIBLE_DEVICES=0 automatically to ensure only GPU 0 is used. If you want to use multiple GPUs, switch to DeepSpeed distributed training.

❌ Training completes but GGUF conversion fails

Cause: The llama.cpp conversion script is not installed, or output_dir has insufficient disk space (GGUF conversion requires an additional 2-3x space).

Solution:

  • Confirm that the llama.cpp dependencies are installed.
  • Confirm that the disk where output_dir resides has enough space (3-4x the original model size).
  • The HF-format model still exists in output_dir and can be loaded with the Transformers engine.

❌ Loss does not converge or stays at 0

Cause: The dataset format is incorrect, or completion_only_loss results in no trainable tokens.

Solution:

  • When using dual-field mode, confirm that the completion field has enough text content (it must not be empty).
  • If the completion is very short (< 10 tokens), try switching to single-field text_field mode.
  • Confirm that max_seq_length is not so small that the completion gets fully truncated.
  • Reduce the learning_rate (e.g., 2e-4 → 1e-4).

❌ Redis connection failed: training history unavailable

Symptom: /training/status/{session_id}/history returns 404, or the log shows a Redis connection warning.

Explanation: Redis is an optional component. If Redis is not started, the system falls back to in-memory storage. After the service restarts, the history records in memory are lost.

Solution: Start the Redis service (redis-server), and set REDIS_URL or REDIS_HOST/REDIS_PORT in .env.

❌ DeepSpeed nvme offload path not writable

Symptom: DeepSpeed training fails, and the log shows an error related to the nvme path.

Solution:

  • Confirm that the offload_folder directory exists and is writable (the system creates it automatically, but you must confirm the parent directory has write permission).
  • Confirm that there is enough disk space (ZeRO-3 Disk offload may require tens of GB).
  • Use an NVMe SSD rather than a traditional HDD (the speed difference is enormous).
