1. 系統架構總覽

Fine-tune Service 採用與 Inference Service 相同的多進程隔離架構。訓練工作在獨立的 Worker Process 中執行，確保 OOM 或崩潰不影響主 FastAPI 服務。

graph TD Client[Client / Frontend] -- HTTP API --> API[FastAPI Server (service/app.py)] subgraph MainProcess [Main Process] API -- calls --> TM[TrainingManager (Singleton)] TM -- delegates --> TPM[TrainingProcessManager] TPM -- stores --> Redis[(Redis / Memory)] MR[ModelRegistry] -- registers finetuned model --> Registry[(models_registry.json)] end subgraph WorkerProcess [Isolated Training Worker Process] TPM -- Queue IPC --> Worker[_training_worker_process] Worker -- 1. Strategy --> Strategy[StrategyFactory] Worker -- 2. Load --> Tokenizer[AutoTokenizer] Worker -- 2. Load --> Dataset[Dataset JSONL] Worker -- 3. Load --> Model[AutoModelForCausalLM + PEFT] Worker -- 4. Train --> Trainer[SFTTrainer / CustomTrainer] Trainer -- 5. Save --> OutputDir[output_dir/] OutputDir -- 6. Convert --> GGUF[GGUF Q4_K_M file] OutputDir -- 7. Register --> MR end Worker -- progress / status --> TPM
重要：訓練進程在任一情況（完成/停止/錯誤）後都會退出 Worker Process，確保 VRAM 完全釋放。下次訓練時會重新啟動乾淨的 Worker。

2. 訓練方法比較

LoRA

Low-Rank Adaptation

  • 只訓練低秩矩陣，參數量極少
  • GPU 需求低
  • 訓練速度快
  • 輸出：adapter 權重檔

QLoRA

Quantized LoRA (4-bit)

  • 基礎模型以 4-bit 量化加載
  • GPU 需求更低（適合消費級 GPU）
  • 稍有精度損失
  • 輸出：adapter 權重檔

Full Parameter

全參數微調

  • 訓練所有參數
  • 需大量 GPU VRAM
  • 效果最佳
  • 通常搭配 DeepSpeed ZeRO
特性 LoRA QLoRA Full
method 值 lora qlora full
建議 GPU VRAM (7B 模型) ≥ 16 GB ≥ 8 GB ≥ 80 GB 或 DeepSpeed
基礎模型量化 FP16/BF16 4-bit NF4 BF16
可訓練參數 ~0.1–1% ~0.1–1% 100%
輸出格式 LoRA adapter LoRA adapter 完整模型權重
DeepSpeed 支援 可選 可選 強烈建議

3. 關鍵 API 與流程

所有訓練相關 API 以 /training/ 為前綴，定義於 service/app.py。

1. 開始訓練

POST /training/start

行為: 同步驗證 + 非同步訓練。

傳入完整 TrainingConfig，驗證通過後立即返回 session_id。實際訓練在 Worker Process 背景執行。

注意： 若 output_dir 已存在檔案，API 返回錯誤（需手動清空）。

2. 查詢訓練狀態

GET /training/status

行為: 同步，輪詢使用。

返回 TrainingStatus：包含 is_training, progress (0–1), current_step, total_steps, loss, current_epoch, status, error。

3. 停止訓練

POST /training/stop

行為: 強制終止 Worker Process（terminate + kill）。

不等待 Worker 優雅退出，直接終止進程並重置狀態。已儲存的 checkpoint 不受影響。

4. 訓練歷史紀錄

GET /training/status/{session_id}/history

返回該訓練 session 的詳細 loss/learning_rate/epoch 紀錄。

資料存於 Redis（若可用）或記憶體，以 session_id 索引。

5. 系統資源歷史

GET /system/resource/{session_id}/history

返回訓練期間的 CPU / GPU / Disk 使用量歷史紀錄，每個 logging_steps 採樣一次。

6. 錯誤詳情

GET /training/error_details

取得完整的訓練錯誤 traceback，包含 is_oom 標誌。格式與 /inference/error_details 一致。

7. 強制清理 GPU

POST /training/force_cleanup_gpu

強制終止 Training Worker Process 並重置狀態，用於 OOM 後 VRAM 未釋放時（不影響 Inference Worker）。

完整訓練流程範例 (Python)

import requests, time

BASE = "http://localhost:8000"

# Step 1: 開始訓練
config = {
    "model_name": "Qwen/Qwen3-4B",
    "method": "lora",
    "dataset_path": "dataset/mydataset.jsonl",
    "output_dir": "./output/qwen3_lora_v1",
    "lora_r": 16,
    "lora_alpha": 32,
    "lora_dropout": 0.05,
    "num_train_epochs": 3,
    "per_device_train_batch_size": 1,
    "gradient_accumulation_steps": 8,
    "learning_rate": 2e-4,
    "max_seq_length": 2048,
    "prompt_field": "prompt",
    "completion_field": "completion",
    "use_sft_trainer": True
}

resp = requests.post(f"{BASE}/training/start", json=config)
session_id = resp.json()["result"]["session_id"]
print(f"Training started: session_id={session_id}")

# Step 2: 輪詢進度
while True:
    status = requests.get(f"{BASE}/training/status").json()
    if status["is_training"]:
        print(f"  Step {status['current_step']}/{status['total_steps']} "
              f"| Progress: {status['progress']:.1%} "
              f"| Loss: {status.get('loss', 'N/A'):.4f}"
              f"| Epoch: {status.get('current_epoch', 0):.2f}")
        time.sleep(10)
    else:
        print(f"Training ended: status={status['status']}")
        if status.get("error"):
            print(f"Error: {status['error']}")
        break

# Step 3: 取得詳細 loss 曲線
history = requests.get(f"{BASE}/training/status/{session_id}/history").json()
for log in history["logs"]:
    print(f"Step {log['step']}: loss={log['loss']:.4f}, lr={log.get('learning_rate')}")

4. 程式碼結構詳解

檔案路徑 角色與功能
service/app.py API 入口。定義 /training/* 路由，呼叫 training_manager。
service/training_manager.py Singleton Facade。暴露 start_training, get_status, stop_training, get_history 等方法給 API 層，並做並發保護（Lock）。
service/training/training_process.py Process 控制核心。包含：
  • TrainingProcessManager: 管理 Worker Process 生命週期、IPC Queue、Status 解析、Redis 歷史讀取。
  • _training_worker_process(): 實際運行在子進程的主循環，接收 start/stop 指令。
service/training/core/strategies.py Strategy Pattern。
  • StrategyFactory: 根據 use_sft_trainer 選擇策略。
  • SFTStrategy: 使用 TRL SFTTrainer（推薦，支援 completion-only loss）。
  • CausalLMStrategy: 使用原生 HF Trainer（手動 tokenize）。
service/training/sft_runner.py SFTTrainer 初始化。建立 CustomSFTTrainer（override save_model 以相容 DeepSpeed + LoRA）和 SFTConfig。
service/training/core/model_loader.py 模型與 Tokenizer 加載。根據方法（LoRA/QLoRA/Full）套用適當的 BitsAndBytesConfig 和 LoraConfig。
service/training/core/dataset_loader.py 資料集加載。讀取 JSON 或 JSONL 格式資料，轉換為 HuggingFace Dataset 物件。
service/training/core/model_saver.py 結果儲存。儲存模型權重、tokenizer，並呼叫 model_registry 將微調後模型登錄為 finetuned 類型。
service/utils/conversion_manager.py GGUF 轉換。訓練完成後自動呼叫 llama.cpp 轉換腳本，生成 Q4_K_M 量化的 GGUF 檔案。
service/configs/deepspeed/ DeepSpeed 配置模板。提供四種預設 ZeRO-3 Offload 組合：
  • zero3_offload_cpu_cpu.json: Optimizer & Param 均 Offload 到 CPU
  • zero3_offload_cpu_disk.json: Optimizer→CPU, Param→Disk
  • zero3_offload_disk_cpu.json: Optimizer→Disk, Param→CPU
  • zero3_offload_disk_disk.json: 均 Offload 到 Disk (NVMe)

5. 訓練流程細節 (Deep Dive)

Worker Process 主流程

1
接收 Start 指令
Worker 從 Request Queue 取出 {"command": "start", "config": {...}, "session_id": "..."}。 檢查 output_dir 是否已存在檔案（若有則回報錯誤，防止覆蓋）。
2
Strategy 選擇
StrategyFactory.get_strategy(config)：
  • use_sft_trainer=True（預設）→ SFTStrategy（使用 TRL SFTTrainer）
  • use_sft_trainer=False → CausalLMStrategy（使用原生 HF Trainer + 手動 tokenize）
同時初始化 TrainingArguments（含 DeepSpeed 配置、BF16、Gradient Checkpointing）。
3
加載資源（依序執行，節省記憶體）
  1. 載入 Tokenizer（AutoTokenizer.from_pretrained）。
  2. 載入 Dataset（JSONL → HuggingFace Dataset）。
  3. 若使用 CausalLMStrategy，此時預先 tokenize 資料集（節省訓練時 CPU 等待）。
  4. 載入 Model（根據方法套用量化設定與 PEFT LoraConfig）。
4
準備 Trainer
strategy.prepare_trainer(model, tokenizer, dataset, training_args)。
  • SFTStrategy：建立 CustomSFTTrainer，自動處理 prompt+completion loss masking（completion_only_loss=True）。
  • CausalLMStrategy：建立 CustomTrainer，使用 DataCollatorForLanguageModeling。
加入 _ProgressCallback，每個 logging_steps 回報 progress 到 Status Queue 並存入 Redis。
5
執行訓練
trainer.train()，期間每個 logging_steps：
  • 推送 {"status": "progress", "step": N, "loss": ..., "epoch": ...} 到 Status Queue。
  • 採樣 CPU/GPU/Disk 資源使用量並存入 Redis。
6
儲存結果
save_training_results(trainer, tokenizer, training_config) 將 adapter 或完整模型存入 output_dir，並自動呼叫 model_registry 登錄為 finetuned 模型。
7
清理資源 + GGUF 轉換
先執行 _cleanup_training_resources()（刪除 Trainer/Model/Tokenizer 物件、GC、清空 CUDA cache），確保 VRAM 釋放後，再呼叫 _convert_training_output_to_q4_k_m() 進行 GGUF 轉換。
8
回報完成並退出
推送 {"status": "completed", "gguf_path": "...", "gguf_quantization": "Q4_K_M"} 到 Status Queue。Worker Process 退出，確保所有資源釋放。

進度狀態機

stateDiagram-v2 [*] --> idle idle --> starting: start_training() starting --> initializing: Worker 接收到指令 initializing --> running: trainer.train() 開始 running --> completed: 訓練成功完成 + GGUF 轉換 running --> stopped: 收到 stop 指令 running --> error: 發生 OOM 或 Exception completed --> idle: 下次訓練 stopped --> idle: 下次訓練 error --> idle: force_cleanup_gpu 後

6. 資料集格式

訓練資料必須是 JSON 或 JSONL 格式（JSONL 為每行一個 JSON 物件）。支援兩種訓練模式，透過 text_field 或 prompt_field/completion_field 選擇。

模式一：text 單欄位模式（全序列 loss）

適用：語言模型預訓練或需要完整序列訓練的場景。設定 text_field: "text"（或對應欄位名）。

// dataset.jsonl（每行一筆）
{"text": "### Question:\n如何設定 Linux 系統時區？\n\n### Answer:\n使用 timedatectl 命令：sudo timedatectl set-timezone Asia/Taipei"}
{"text": "### Question:\n什麼是 Docker volume？\n\n### Answer:\nDocker volume 是用於持久化容器資料的機制..."}
{"text": "這是一段需要模型學習的文字內容..."}

模式二：prompt + completion 雙欄位模式（completion-only loss）

適用：指令微調（Instruction Fine-tuning）。SFTTrainer 會自動遮蔽 prompt 的 loss，只計算 completion 部分的 loss。

設定 prompt_field: "prompt", completion_field: "completion"。

// mydataset.jsonl（每行一筆）
{"prompt": "你是一個 Linux 系統管理員。\n如何查看系統記憶體使用量？", "completion": "使用 free -h 命令可以查看記憶體使用量：\n```\nfree -h\n```\n其中 -h 表示以人類可讀格式顯示..."}
{"prompt": "解釋 Python 中的 GIL 是什麼", "completion": "GIL（Global Interpreter Lock）是 CPython 的一個機制..."}
{"prompt": "如何在 Git 中撤銷最後一個 commit？", "completion": "有幾種方式：\n1. 保留變更：git reset --soft HEAD~1\n2. 丟棄變更：git reset --hard HEAD~1"}
推薦選擇： 指令微調建議使用雙欄位模式 + use_sft_trainer: true，系統會自動啟用 completion_only_loss=True，讓模型只學習回答部分，訓練效率更高。

資料集品質建議

建議項目說明
最小資料量 建議至少 500–1000 筆，資料質量比數量更重要
序列長度 控制在 max_seq_length 以內，超過會被截斷；建議 95% 的資料不超過設定長度
資料清理 移除重複資料、確保 UTF-8 編碼正確、避免空白 prompt/completion
格式一致性 同一資料集中 prompt 格式需一致（如是否加 system prompt）

7. 配置與參數說明

完整 TrainingConfig 說明

{
  // === 必填 ===
  "model_name": "Qwen/Qwen3-4B",        // Model Registry 中的 label 或 HF model ID
  "method": "lora",                       // 訓練方法: lora / qlora / full
  "dataset_path": "dataset/train.jsonl",  // 資料集路徑（相對或絕對路徑）
  "output_dir": "./output/my_model_v1",   // 輸出目錄（必須為空目錄）

  // === LoRA/QLoRA 專用 ===
  "lora_r": 16,                           // LoRA 秩（越大→參數越多，效果越好但更慢）
  "lora_alpha": 32,                       // LoRA scaling（通常 = lora_r * 2）
  "lora_dropout": 0.05,                   // Dropout（防止過擬合）
  "lora_target_modules": [               // 要訓練的模組（留空/null = 自動偵測線性層，偵測失敗才 fallback 到此列表）
    "q_proj", "k_proj", "v_proj", "o_proj",
    "gate_proj", "up_proj", "down_proj"
  ],

  // === 訓練超參數 ===
  "num_train_epochs": 3,                 // 訓練輪數
  "per_device_train_batch_size": 1,      // 每次 batch 大小（記憶體不足時降為 1）
  "gradient_accumulation_steps": 8,      // 累積梯度步數（有效 batch = batch * accum）
  "learning_rate": 2e-4,                 // 學習率（LoRA/QLoRA 常用 1e-4 ~ 3e-4）
  "warmup_steps": 100,                   // 學習率預熱步數
  "max_seq_length": 2048,               // 最大序列長度（超過截斷）

  // === 資料集欄位配置（二擇一）===
  // 模式一（全序列 loss）：
  "text_field": "text",
  // 模式二（completion-only loss，推薦用於指令微調）：
  "prompt_field": "prompt",
  "completion_field": "completion",

  // === 儲存設定 ===
  "logging_steps": 10,                   // 每幾步回報一次 loss
  "save_steps": 500,                     // 每幾步儲存一次 checkpoint
  "save_total_limit": 2,                 // 最多保留幾個 checkpoint
  "save_tokenizer": true,                // 是否儲存 tokenizer

  // === Trainer 選擇 ===
  "use_sft_trainer": true,               // 使用 TRL SFTTrainer（推薦）
  "packing": false,                      // 是否啟用序列 packing
  "gradient_checkpointing": true,        // 梯度 checkpointing（節省 VRAM）

  // === DeepSpeed（可選）===
  "use_deepspeed": false,
  "deepspeed_profile": "zero3_offload_cpu_cpu",  // 使用內建 profile
  // 或
  "deepspeed_config": "./my_deepspeed.json",       // 使用自訂路徑
  "offload_folder": "./deepspeed_offload"          // ZeRO offload 目錄
}

場景methodlora_rbatch_sizeaccumulationlearning_rate

8. DeepSpeed Offload 配置

當 GPU VRAM 不足以容納完整模型（通常用於 Full parameter 訓練或大型模型 LoRA），可啟用 DeepSpeed ZeRO-3 Offload 將部分資料卸載至 CPU RAM 或 NVMe Disk。

注意： DeepSpeed 會顯著延長訓練時間（CPU offload 慢 3-5 倍，Disk offload 慢 5-20 倍），建議優先嘗試 QLoRA + 較小 batch size，只在必要時才啟用 DeepSpeed。

四種內建 Profile

Profile 名稱 Optimizer Offload Param Offload 適用場景
zero3_offload_cpu_cpu CPU RAM CPU RAM 有大量 CPU RAM（>64GB），最常用
zero3_offload_cpu_disk CPU RAM NVMe Disk CPU RAM 不足，但有快速 NVMe
zero3_offload_disk_cpu NVMe Disk CPU RAM 特殊場景
zero3_offload_disk_disk NVMe Disk NVMe Disk 極低記憶體環境，需高速 NVMe

啟用 DeepSpeed 的配置範例

{
  "model_name": "Qwen/Qwen3-14B",
  "method": "lora",
  "dataset_path": "dataset/train.jsonl",
  "output_dir": "./output/qwen3_14b_lora",
  "use_deepspeed": true,
  "deepspeed_profile": "zero3_offload_cpu_cpu",  // 使用內建 profile
  "offload_folder": "./deepspeed_offload",         // 若 profile 有 nvme，此路徑覆蓋
  "gradient_checkpointing": true,
  "lora_r": 16,
  "lora_alpha": 32,
  "num_train_epochs": 3
}
DeepSpeed + LoRA 特別處理： ZeRO-3 的 save_model 預設行為與 LoRA adapter 不相容（可能 OOM 或 FileExists 錯誤）。系統已透過 CustomSFTTrainer 和 CustomTrainer 覆蓋 save_model，使用 GatheredParameters 安全收集 adapter 參數後儲存，此問題已自動處理。

DeepSpeed 相關環境設定

Worker Process 啟動時自動設定以下環境變數：

WORLD_SIZE=1
RANK=0
LOCAL_RANK=0
MASTER_ADDR=127.0.0.1
MASTER_PORT=29500
DS_SKIP_CUDA_CHECK=1          # 跳過 CUDA 版本檢查
PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True  # 減少記憶體碎片
CUDA_VISIBLE_DEVICES=0        # 強制使用 GPU 0（防止多 GPU PEFT 問題）

9. 訓練後流程 (Post-Training)

自動化後處理：訓練成功完成後，系統會自動執行以下步驟，無需手動操作：

自動 GGUF 轉換

  1. 訓練完成 + 模型儲存至 output_dir。
  2. 清理訓練資源（VRAM 釋放）。
  3. 呼叫 conversion_manager.convert_and_quantize()：
    • 先將 HF 格式模型轉換為 GGUF F16。
    • 再量化為 Q4_K_M 格式（體積更小、推理更快）。
  4. GGUF 檔案存於 output_dir 內。
  5. /training/status 回傳 gguf_path 和 gguf_quantization。

模型自動登錄至 Registry

訓練完成後，微調模型自動以 finetuned 類型登錄至 models_registry.json，可在下列 API 查詢：

GET /config/models
// 回傳中包含：
{
  "finetuned_models": [
    {
      "label": "my_model_v1",
      "output_dir": "./output/my_model_v1",
      "base_model": "Qwen/Qwen3-4B",
      "method": "lora",
      ...
    }
  ]
}

使用微調模型進行推理

訓練後的模型可直接透過 Inference API 加載（需指定 model_path 為 output_dir）：

POST /inference/load_model
{
  "model_name": "my_model_v1",        // Registry label
  "model_path": "./output/my_model_v1", // 微調輸出目錄
  "engine": "transformers",
  "quantization": "none"              // adapter 已是低精度，不需額外量化
}

或使用自動轉換的 GGUF 模型（推薦生產環境）：

POST /inference/load_model
{
  "model_name": "my_model_v1_gguf",
  "model_path": "./output/my_model_v1/my_model_v1-Q4_K_M.gguf",
  "engine": "llama_server",
  "n_gpu_layers": -1
}

10. 常見問題與處理

❌ Error: Output directory already exists and contains files

原因: output_dir 已有先前訓練的殘留檔案。

解決方案:

  • 手動清空或刪除 output_dir。
  • 或改用新的目錄名稱（如加上版本號：my_model_v2）。
系統不會自動清空輸出目錄，以防意外覆蓋先前訓練成果。

❌ CUDA Out of Memory (OOM) during training

症狀: 訓練時崩潰，/training/error_details 的 is_oom=true。

解決方案（依優先順序）:

  1. 呼叫 POST /training/force_cleanup_gpu 確保 VRAM 釋放。
  2. 改用 method: "qlora"（4-bit 量化基礎模型）。
  3. 降低 per_device_train_batch_size 至 1，增加 gradient_accumulation_steps。
  4. 降低 max_seq_length（2048 → 1024 → 512）。
  5. 啟用 DeepSpeed CPU Offload：use_deepspeed: true, deepspeed_profile: "zero3_offload_cpu_cpu"。

❌ Training Error: RuntimeError / CUBLAS_STATUS_ALLOC_FAILED

原因: 多 GPU 環境下 PEFT 的設備分配問題。

解決方案: Worker Process 已自動設定 CUDA_VISIBLE_DEVICES=0，確保只使用 GPU 0。若使用多 GPU，需改用 DeepSpeed 分散式訓練。

❌ Training 完成但 GGUF 轉換失敗

原因: llama.cpp 轉換腳本未安裝或 output_dir 磁碟空間不足（GGUF 轉換需要額外 2-3 倍空間）。

解決方案:

  • 確認已安裝 llama.cpp 相依套件。
  • 確認 output_dir 所在磁碟有足夠空間（原模型大小的 3-4 倍）。
  • HF 格式模型仍存在於 output_dir，可用 Transformers 引擎加載。

❌ Loss 不收斂或一直為 0

原因: 資料集格式錯誤，或 completion_only_loss 導致沒有可訓練的 token。

解決方案:

  • 使用雙欄位模式時，確認 completion 欄位有足夠的文字內容（不可為空）。
  • 若 completion 很短（< 10 tokens），嘗試改用 text_field 單欄位模式。
  • 確認 max_seq_length 不要太小導致 completion 被截斷完。
  • 減小 learning_rate（如 2e-4 → 1e-4）。

❌ Redis 連線失敗：訓練歷史不可用

症狀: /training/status/{session_id}/history 回傳 404，或 log 顯示 Redis 連線警告。

說明: Redis 是選配元件。若未啟動 Redis，系統 fallback 至記憶體儲存。服務重啟後，記憶體中的歷史紀錄會清失。

解決方案: 啟動 Redis 服務（redis-server），並在 .env 中設定 REDIS_URL 或 REDIS_HOST/REDIS_PORT。

❌ DeepSpeed nvme offload 路徑不可寫

症狀: DeepSpeed 訓練失敗，log 顯示 nvme 路徑相關錯誤。

解決方案:

  • 確認 offload_folder 目錄存在且可寫（系統會自動建立，但需確認上層目錄有寫入權限）。
  • 確認磁碟空間足夠（ZeRO-3 Disk offload 可能需要數十 GB）。
  • 使用 NVMe SSD 而非傳統 HDD（速度差距極大）。
