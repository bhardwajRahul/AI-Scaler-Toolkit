Trusta AST Inference Service User Manual

This document explains in detail the architecture of the inference service, the API flow, and the code structure. It is intended for team training.

Applicable version: supports three inference engines (Transformers / vLLM / llama-server); the primary chat endpoint has been upgraded to an OpenAI-compatible API.

This is the English version. For the Traditional Chinese version, see inference_manual.html.

1. Architecture Overview

This service is a high-performance LLM inference server built on FastAPI. It supports multiple inference engines and, in order to avoid the Python GIL limitation and manage GPU VRAM effectively, adopts a Multi-process Isolation architecture.

Core Design Patterns: Singleton Manager & Process Isolation

The main components are managed through the singleton pattern to ensure global state consistency, while model loading and inference work are isolated in a dedicated Worker Process.

  • Main Process (FastAPI): Handles HTTP requests, route dispatch, and session management.
  • Worker Process (Model Inference): Dedicated to PyTorch model loading, holding the weights, and running inference computation.
  • Advantage: On OOM the Worker Process can be safely terminated and restarted, ensuring VRAM is fully released without affecting the main service.
graph TD Client[Client / Frontend] -- HTTP/OpenAI API --> API[FastAPI Server (service/app.py)] subgraph MainProcess [Main Process] API -- calls --> MM[ModelManager (Singleton)] SM[SessionManager] -- loads/saves --> Redis[(Redis/Memory)] MR[ModelRegistry] -- reads/writes --> Registry[(models_registry.json)] end subgraph WorkerProcess [Isolated Worker Process] MM -- Queue IPC --> MIP[ModelInferenceProcess] MIP -- uses --> Engine{Engine Selector} Engine -- Transformers --> TF[HF Transformer Model] Engine -- vLLM --> VLLM[vLLM Server subprocess] Engine -- llama_server --> LLAMA[llama-server subprocess] end API -- reads status --> MM

2. Inference Engines

Different inference engines can be selected through the InferenceConfig.engine field. Each engine is suited to a different use case.

Engine engine value Use case Notes
Transformers transformers Default engine; supports int4/int8 quantization, CPU offload, and disk offload Broadest compatibility; ideal for low-resource environments
vLLM vllm High-throughput production environments; native concurrent multi-request handling; supports AWQ/GPTQ quantization Requires the vllm package; slower to start but high throughput
llama-server llama_server GGUF-format models; ideal for deploying quantized models Requires a path to a .gguf file; uses a llama.cpp subprocess

Engine-Specific Configuration Fields

vLLM Engine Specific Parameters

  • vllm_gpu_memory_utilization: GPU memory utilization (default 0.8)
  • vllm_max_model_len: Maximum context length
  • vllm_dtype: Data type (default auto)
  • vllm_quantization: Quantization method (awq/gptq/fp8)
  • vllm_tensor_parallel_size: Tensor parallelism degree (multi-GPU)
  • vllm_cpu_offload_gb: CPU offload size (GB)
  • vllm_mm_image_limit: Maximum multimodal image count

llama-server Specific Parameters

  • n_gpu_layers: Number of GPU layers (-1 = all)
  • n_ctx: Context length (default 4096)
  • n_batch: Batch size (default 512)
  • llama_server_np: Number of parallel generation slots
  • llama_server_port: Port used by the subprocess (default 5001)
  • llama_server_mmproj: Multimodal projector path
  • llama_server_auto_start: Automatically start the subprocess (default true)

3. Key APIs and Flows

All API entry points are defined in service/app.py. The primary chat endpoint has been upgraded to the OpenAI-compatible format.

Important: The POST /inference/chat endpoint has been marked Deprecated and returns HTTP 410. Please switch to POST /v1/chat/completions across the board.

Inference-Related APIs

1. Load Model

POST /inference/load_model

Behavior: Asynchronous background loading.

The API returns a loading status immediately; poll via /inference/status. If a model with the same configuration is already loaded, it returns already_loaded; if a different model is already loaded, it returns HTTP 409.

Key parameters: model_name, engine, quantization, device_map, max_memory.

2. Unload Model

POST /inference/unload_model

Behavior: Synchronous.

Terminates the Worker Process and releases all GPU/CPU memory.

3. Check Status

GET /inference/status

Behavior: Synchronous.

Returns whether loading is complete, the current engine, any OOM error message, and device map allocation statistics.

4. OpenAI-Compatible Chat Primary Endpoint

POST /v1/chat/completions

Behavior: Supports both streaming (SSE) and non-streaming, fully compatible with the OpenAI API format.

Supports session memory, tool calls, and multimodal input.

Key parameters: messages, stream, session_id, enable_thinking, tools.

5. Legacy Chat Deprecated

POST /inference/chat

This endpoint is deprecated and returns HTTP 410 when called. Please use /v1/chat/completions instead.

6. Estimate Memory

POST /inference/estimate_memory

Behavior: Pure computation; does not load the model.

Estimates the GPU requirements under each offload strategy, helping you judge whether the hardware is sufficient.

Also available via GET: /inference/estimate_memory/{model_name}?quantization=int4

7. Stop Generation

POST /inference/stop_generation

Stops the generation task currently in progress. Pass request_id to stop a specific request precisely, or session_id to stop generation for that session.

8. Force GPU Cleanup

POST /inference/force_cleanup_gpu

Forcibly terminates the Worker Process and restarts a clean process, used when VRAM has not been released after an OOM.

9. Error Details

GET /inference/error_details

Retrieves the full error traceback, useful for diagnosing model loading failures.

10. Clean Up KV Cache

POST /inference/cleanup_generation_memory

Cleans up memory accumulated by the KV cache without unloading the model. Suitable after long conversations or before an OOM retry.

OpenAI-Compatible API Usage Example

# 串流對話 (Python)
import httpx, json

url = "http://localhost:8000/v1/chat/completions"
payload = {
    "model": "Qwen/Qwen3-4B",
    "messages": [
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "請解釋量子力學的基本概念"}
    ],
    "stream": True,
    "temperature": 0.7,
    "max_tokens": 1024,
    "session_id": "user-123",          # 啟用對話記憶
    "enable_thinking": False           # 關閉 thinking 模式
}

with httpx.Client(timeout=300) as client:
    with client.stream("POST", url, json=payload) as resp:
        for line in resp.iter_lines():
            if line.startswith("data: ") and line != "data: [DONE]":
                chunk = json.loads(line[6:])
                delta = chunk["choices"][0]["delta"].get("content", "")
                print(delta, end="", flush=True)

4. Code Structure in Detail

Understanding the project structure helps you locate problems quickly.

File path Role and function
service/app.py Entry point. Defines all API routes, CORS, lifecycle management, and the OpenAI-compatibility layer (session assembly, streaming handling).
service/model_manager.py Mediator. A singleton that coordinates communication between the API layer and the Worker Process. Provides methods such as start_loading, generate_stream, generate, and stop_generation.
service/inference/model_inference_process.py Process controller. Manages the multiprocessing.Process, controlling the Worker lifecycle and IPC Queue communication.
service/inference/engines/ Engine implementations.
  • base_engine.py: Abstract base class
  • transformers_engine.py: HF Transformers engine
  • vllm_engine.py: vLLM engine (subprocess management)
  • llama_server_engine.py: llama-server engine
service/inference/generate/generator_worker.py Generation logic. Contains handle_generate_request and handle_generate_stream_request, which call model.generate() and handle the Streamer output.
service/inference/load_model/peft_loader.py PEFT loader. Supports loading fine-tuned LoRA/QLoRA adapter models (merged into the base model or loaded as a PeftModel).
service/config_models.py Data models. Defines all Pydantic request/response schemas, including InferenceConfig, OpenAIChatCompletionRequest, TrainingConfig, and others.
service/session_manager.py Session management. Manages multi-turn conversation history, supporting Redis storage (when Redis is available) or in-memory storage (fallback).
service/model_registry.py Model registry. Manages the list of available models (base/finetuned/llama_gguf), stored in service/configs/models_registry.json.
service/download_manager.py Download management. Asynchronously downloads models from the HuggingFace Hub and automatically registers them in the Model Registry.
service/utils/conversion_manager.py Format conversion. Converts HF models to GGUF format (invokes the llama.cpp conversion script).
service/inference/memory_estimator.py Memory estimation. Estimates VRAM requirements under each offload strategy based on model size and quantization settings.
service/settings.py Environment configuration. Reads settings from .env such as HF_HOME, Redis, and the service port.

5. Inference Process Details (Deep Dive)

Step 1: Model Loading Flow (Load Flow)

  1. The client sends POST /inference/load_model (with engine, model_name, quantization, etc.).
  2. app.py checks whether a model with the same configuration is already running (if so, already_loaded), or whether a different model is running (in which case HTTP 409 requires unloading first).
  3. app.py calls model_manager.start_loading(config).
  4. ModelManager pushes the load command onto the Request Queue (IPC).
  5. The Worker Process selects the corresponding engine based on engine:
    • Transformers: Loads using BitsAndBytesConfig, AutoModelForCausalLM, and a device map.
    • vLLM: Starts a vllm serve subprocess and waits for the health check to pass.
    • llama-server: Starts a llama-server subprocess and waits for /health to be ready.
  6. The Worker continuously reports progress through the Status Queue (loading → ready).

Step 2: OpenAI-Compatible Chat Flow

Note: /v1/chat/completions is responsible for session management and prompt assembly in the main process, while the inference itself runs in the Worker Process.
  1. The client sends POST /v1/chat/completions (with messages, stream=True, session_id).
  2. app.py performs prompt assembly:
    • If a session_id is present, it retrieves the history via session_manager (up to 6 messages, roughly 3 turns of conversation).
    • It parses enable_thinking and applies model-aware generation parameters (for example, Qwen3.5 automatically adjusts temperature/top_p/repetition_penalty).
  3. If the request contains tools or tool_calls, it takes the tool passthrough path (bypassing session management and passing through directly).
  4. app.py calls model_manager.generate_stream() to obtain an iterator.
  5. The Worker Process runs the generation, using TextIteratorStreamer to push tokens one by one onto the Data Queue.
  6. Through SSE, app.py wraps the Data Queue content into the OpenAI chunk format in real time and returns it to the frontend.
  7. After generation completes, app.py appends this turn's user/assistant exchange to session_manager.

Qwen3.5 Model-Aware Parameter Auto-Adjustment

Special behavior: When the loaded model is detected to be part of the Qwen3.5 series (the model name contains "qwen3.5"), the system automatically forces an override of the generation parameters:
  • temperature is forced to 1.0
  • top_p is forced to 0.95
  • top_k is forced to 20
  • repetition_penalty is forced to 1.0
  • enable_thinking is forced to false
This is done to prevent the Qwen3.5 MoE architecture from producing multilingual garbled output under non-official parameters.

7. Model Registry

The model list is stored in service/configs/models_registry.json and is divided into three types: base (HF models), finetuned (after fine-tuning), and llama_gguf (GGUF).

Query Available Models

GET /config/models

Returns the list of all registered base models, fine-tuned models, and GGUF models.

Download Model

POST /config/models/download

Downloads a model from the HuggingFace Hub and automatically registers it in the registry. Runs asynchronously and returns a task_id; you can query progress via GET /config/models/download/{task_id}.

{
  "model_id": "Qwen/Qwen3-4B",
  "label": "qwen3_4b",
  "force_download": false
}

Delete Model

DELETE /config/models/{label}

Removes the model from the registry. Adding ?delete_files=true also deletes the local files (including the entire HF cache directory).

Convert to GGUF

POST /config/models/convert

Converts an HF model to GGUF format, running asynchronously. Supports outtype: auto/f16/bf16/q8_0/f32.

8. Troubleshooting

❌ Error: Out of Memory (OOM)

Symptom: The Worker Process crashes and /inference/status shows loading_error.

Solutions:

  • Call POST /inference/force_cleanup_gpu to forcibly clean up VRAM.
  • Lower the max_memory GPU limit to force more layers to offload to the CPU.
  • Switch to a higher quantization level (int8 → int4).
  • Use vllm_cpu_offload_gb (vLLM engine).
  • Call /inference/estimate_memory first to assess requirements before deciding on the configuration.

❌ HTTP 409: Model Already Loaded

Symptom: Receiving a 409 when trying to load a different model.

Solution: Call POST /inference/unload_model first, then load again.

❌ HTTP 410: Deprecated Endpoint

Symptom: Calling POST /inference/chat returns a 410.

Solution: Switch to POST /v1/chat/completions, which is compatible with the OpenAI API format.

❌ Inference Timeout

Symptom: Generation takes too long, causing the connection to drop.

Solutions:

  • Lower the max_tokens parameter (which maps to max_new_tokens).
  • Raise total_timeout in the request (default 300 seconds).
  • Check whether too many layers are on CPU offload (CPU inference is extremely slow).

❌ Tokenizer / Template Error

Symptom: The response content contains strange tags or formatting errors.

Solutions:

  • Confirm that trust_remote_code: true is enabled.
  • Check whether the model requires a special chat template (in vLLM you can specify a custom .jinja file via vllm_chat_template).
  • For Qwen3.5, the system already disables thinking mode automatically; if you still have issues, confirm that the API request does not forcibly set enable_thinking: true.

❌ vLLM Startup Failure / Port Conflict

Symptom: The vLLM engine fails to load and the log shows the port is in use.

Solutions:

  • On service startup, sweep_stale_vllm_processes() runs automatically to clean up leftover processes.
  • Manually check port usage: lsof -i :5000 (the vLLM default port VLLM_PORT), and if necessary kill -9 {PID}.
  • Restart the FastAPI service and try loading again.
