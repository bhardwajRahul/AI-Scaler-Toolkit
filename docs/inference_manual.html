1. 系統架構總覽

本服務是一個基於 FastAPI 的高性能 LLM 推理伺服器，專為處理大型模型設計。為了避免 Python 的 GIL 限制以及更有效地管理 GPU 記憶體 (VRAM)，我們採用了多進程隔離 (Multi-process Isolation) 架構。

核心設計模式：Singleton Manager & Process Isolation

主要組件透過單例模式 (Singleton) 管理，確保全域狀態一致性，而繁重的模型加載與推理工作則被隔離在獨立的 Worker Process 中。

  • 主進程 (FastAPI): 處理 HTTP 請求、路由分發、背景任務排程。
  • Worker Process (Model Inference): 專職負責 PyTorch 模型加載、權重持有及推理計算。
  • 優勢: 當發生 OOM (Out of Memory) 時，可以安全地終止 Worker Process 並重啟，確保 VRAM 被完全釋放，不會導致整個 Web Service 崩潰。

2. 關鍵 API 與流程

所有的 API 入口點定義在 service/app.py 中。以下是您最常交互的幾個端點。

1. 加載模型 (Load Model)

POST /inference/load_model

行為: 非同步。

API 收到請求後會立即返回 loading 狀態，實際載入在背景執行。需透過 /status 輪詢進度。

關鍵參數: model_name, quantization (int4/int8), offload_folder。

2. 檢查狀態 (Check Status)

GET /inference/status

行為: 同步。

返回當前模型是否載入完成、是否有錯誤 (OOM)、以及目前使用的 Device Map。

3. 文字對話 (Chat)

POST /inference/chat

行為: 支援串流 (SSE) 與非串流。

這是主要的推理接口。支援 Session 記憶、RAG 檢索增強。

關鍵參數: message, stream=true, session_id。

4. 記憶體估算 (Estimate)

POST /inference/estimate_memory

行為: 純計算，不載入模型。

在實際載入前，幫助使用者判斷硬體是否足夠執行特定量化設定的模型。

3. 程式碼結構詳解

理解專案結構有助於快速定位問題。

檔案路徑 角色與功能
service/app.py 入口點。定義 API 路由、請求驗證、CORS 設定以及生命週期管理。
service/model_manager.py 中介者。Singleton 類別，負責協調 API 層與底層 Worker 進程的通訊。提供 start_loading, generate_stream 等高層方法。
service/inference/model_inference_process.py 核心引擎。管理 multiprocessing.Process。
  • ModelInferenceProcess: 進程控制器 (Start/Stop)。
  • _model_worker_process: 實際運行在子進程的 Loop，監聽指令 (Load/Generate)。
service/inference/generate/generator_worker.py 生成邏輯。包含 handle_generate_request 和 handle_generate_stream_request。負責調用 HF model.generate() 並處理 Streamer 輸出。
service/config_models.py 資料模型。定義 Pydantic Models (Request/ResponseSchemas)，如 InferenceConfig, ChatRequest。

4. 推理流程細節 (Deep Dive)

Step 1: 模型加載 (Load Flow)

  1. Client 發送 POST /inference/load_model。
  2. app.py 呼叫 model_manager.start_loading(config)。
  3. ModelManager 將 {command: "load", config: ...} 推送到 Request Queue。
  4. Worker Process 從 Queue 取出指令，開始執行：
    • 使用 AutoTokenizer.from_pretrained 載入 Tokenizer。
    • 根據 Config 計算 BitsAndBytesConfig (若使用量化)。
    • 計算 Smart Device Map (若使用 auto device map)。
    • 執行 AutoModelForCausalLM.from_pretrained 載入權重。
  5. 載入過程中，Worker 持續將進度推送到 Status Queue。
  6. 載入完成，Worker 推送 status: ready。

Step 2: 對話生成 (Chat Generation Flow)

特別注意： 推理過程完全在 Worker Process 內進行，主進程只負責轉發結果。
  1. Client 發送 POST /inference/chat (設 stream=True)。
  2. app.py 進行預處理：
    • 通過 session_manager 獲取歷史對話。
    • 通過 rag_manager 檢索相關文檔 (若啟用 RAG)。
    • 使用 apply_chat_template 組合最終 Prompt。
  3. app.py 呼叫 model_manager.generate_stream() 獲得一個迭代器。
  4. ModelManager 向 Request Queue 發送 generate_stream 指令。
  5. Worker Process 接到指令：
    • 調用 generator_worker.handle_generate_stream_request。
    • 啟動一個獨立 Thread 執行 model.generate()。
    • 使用 TextIteratorStreamer 捕獲生成的 Token。
    • 將 Token 即時推送到 Data Queue。
  6. app.py 透過 SSE (Server-Sent Events) 將 Data Queue 內容即時回傳給前端。

5. 配置與參數說明

InferenceConfig (載入模型用)

{
  "model_name": "Qwen/Qwen2.5-14B-Instruct", // HF 模型 ID 或路徑
  "quantization": "int4",                    // 選項: "none", "int8", "int4", "nf4"
  "device_map": "auto",                      // 通常設為 auto
  "max_memory": {                            // 限制各裝置記憶體用量
    "0": "22GiB",                            // GPU 0
    "cpu": "64GiB"                           // CPU Offload 緩衝
  },
  "offload_folder": "offload_dir"            // Disk Offload 路徑
}

ChatRequest (對話用)

{
  "message": "你好，請解釋一下量子力學。",
  "stream": true,
  "temperature": 0.7,
  "max_new_tokens": 2048,
  "session_id": "user-123",    // 用於保持對話記憶
  "use_rag": true,             // 是否啟用知識庫檢索
  "system_prompt": "You are a helpful assistant."
}

6. 常見問題與處理

❌ Error: Out of Memory (OOM)

症狀: Worker Process 崩潰，API 返回 OOM 錯誤。

解決方案:

  • 調低 max_memory 中的 GPU 限制，強制更多層 Offload 到 CPU。
  • 使用更高的量化等級 (如 int8 -> int4)。
  • 使用 POST /inference/force_cleanup_gpu 強制重啟 Worker。

❌ Inference Timeout

症狀: 生成時間過長導致連線中斷。

解決方案:

  • 檢查是否觸發了 CPU Offload (CPU 推理極慢)。
  • 調整 max_new_tokens 參數。
  • 調整 config.total_timeout。

❌ Tokenizer Issue / Template Error

症狀: 回應內容包含奇怪的標籤或格式錯誤。

解決方案:

  • 確認 trust_remote_code=True 是否啟用 (某些模型需要)。
  • 檢查 apply_chat_template 的實作是否兼容該模型。
