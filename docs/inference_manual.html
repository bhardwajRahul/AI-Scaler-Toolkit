Trusta AST Inference Service 教學手冊

本文檔詳細說明了推理服務的架構、API 流程以及程式碼結構，適用於團隊教育訓練。

適用版本：支援三種推理引擎 (Transformers / vLLM / llama-server)，主要對話端點已升級為 OpenAI 相容 API。

1. 系統架構總覽

本服務是一個基於 FastAPI 的高性能 LLM 推理伺服器，支援多種推理引擎，並為了避免 Python GIL 限制及有效管理 GPU VRAM，採用多進程隔離 (Multi-process Isolation) 架構。

核心設計模式：Singleton Manager & Process Isolation

主要組件透過單例模式管理，確保全域狀態一致性，而模型加載與推理工作則隔離在獨立的 Worker Process 中。

  • 主進程 (FastAPI): 處理 HTTP 請求、路由分發、Session 管理。
  • Worker Process (Model Inference): 專職負責 PyTorch 模型加載、權重持有及推理計算。
  • 優勢: OOM 時可安全終止 Worker Process 並重啟，確保 VRAM 完全釋放，不影響主服務。
graph TD Client[Client / Frontend] -- HTTP/OpenAI API --> API[FastAPI Server (service/app.py)] subgraph MainProcess [Main Process] API -- calls --> MM[ModelManager (Singleton)] SM[SessionManager] -- loads/saves --> Redis[(Redis/Memory)] MR[ModelRegistry] -- reads/writes --> Registry[(models_registry.json)] end subgraph WorkerProcess [Isolated Worker Process] MM -- Queue IPC --> MIP[ModelInferenceProcess] MIP -- uses --> Engine{Engine Selector} Engine -- Transformers --> TF[HF Transformer Model] Engine -- vLLM --> VLLM[vLLM Server subprocess] Engine -- llama_server --> LLAMA[llama-server subprocess] end API -- reads status --> MM

2. 推理引擎說明

透過 InferenceConfig.engine 欄位可選擇不同的推理引擎，各引擎適合不同的使用場景。

引擎 engine 值 適用情境 備注
Transformers transformers 預設引擎，支援 int4/int8 量化、CPU Offload、Disk Offload 相容性最廣，適合低資源環境
vLLM vllm 高吞吐量生產環境，原生多請求並行，支援 AWQ/GPTQ 量化 需要安裝 vllm 套件，啟動較慢但吞吐量高
llama-server llama_server GGUF 格式模型，適合量化後部署 需提供 .gguf 檔案路徑，使用 llama.cpp 子程序

引擎專用配置欄位

vLLM 引擎 專用參數

  • vllm_gpu_memory_utilization: GPU 記憶體使用率 (預設 0.8)
  • vllm_max_model_len: 最大 context 長度
  • vllm_dtype: 資料類型 (預設 auto)
  • vllm_quantization: 量化方式 (awq/gptq/fp8)
  • vllm_tensor_parallel_size: 張量並行數量 (多 GPU)
  • vllm_cpu_offload_gb: CPU Offload 大小 (GB)
  • vllm_mm_image_limit: 多模態圖片上限

llama-server 專用參數

  • n_gpu_layers: GPU 層數 (-1=全部)
  • n_ctx: 上下文長度 (預設 4096)
  • n_batch: 批次大小 (預設 512)
  • llama_server_np: 平行生成槽位數
  • llama_server_port: 子程序使用的 Port (預設 5001)
  • llama_server_mmproj: 多模態 projector 路徑
  • llama_server_auto_start: 自動啟動子程序 (預設 true)

3. 關鍵 API 與流程

所有 API 入口點定義在 service/app.py 中。主要對話端點已升級為 OpenAI 相容格式。

重要：POST /inference/chat 端點已標記為 Deprecated，回傳 HTTP 410。請統一改用 POST /v1/chat/completions。

推理相關 API

1. 加載模型 (Load Model)

POST /inference/load_model

行為: 非同步背景加載。

API 立即返回 loading 狀態，需透過 /inference/status 輪詢。若同配置已加載，返回 already_loaded；若不同模型已加載，返回 HTTP 409。

關鍵參數: model_name, engine, quantization, device_map, max_memory。

2. 卸載模型 (Unload Model)

POST /inference/unload_model

行為: 同步。

終止 Worker Process 並釋放所有 GPU/CPU 記憶體。

3. 檢查狀態 (Check Status)

GET /inference/status

行為: 同步。

返回是否載入完成、當前引擎、OOM 錯誤訊息、Device Map 分配統計。

4. OpenAI 相容對話 主要端點

POST /v1/chat/completions

行為: 支援串流 (SSE) 與非串流，完全相容 OpenAI API 格式。

支援 Session 記憶、Tool Call、多模態。

關鍵參數: messages, stream, session_id, enable_thinking, tools。

5. 舊版對話 已棄用

POST /inference/chat

此端點已廢棄，呼叫後返回 HTTP 410。請改用 /v1/chat/completions。

6. 記憶體估算 (Estimate)

POST /inference/estimate_memory

行為: 純計算，不載入模型。

估算各 Offload 策略下的 GPU 需求，幫助判斷硬體是否足夠。

也可用 GET: /inference/estimate_memory/{model_name}?quantization=int4

7. 停止生成

POST /inference/stop_generation

停止當前正在進行的生成任務。可帶 request_id 精準停止特定請求，或帶 session_id 停止該會話的生成。

8. 強制清理 GPU

POST /inference/force_cleanup_gpu

強制終止 Worker Process 並重啟乾淨進程，用於 OOM 後 VRAM 未釋放時。

9. 錯誤詳情

GET /inference/error_details

取得完整的錯誤 traceback，適用模型加載失敗後的診斷。

10. 清理 KV Cache

POST /inference/cleanup_generation_memory

清理 KV cache 累積的記憶體，不卸載模型。適合長對話後或 OOM 重試前使用。

OpenAI 相容 API 使用範例

# 串流對話 (Python)
import httpx, json

url = "http://localhost:8000/v1/chat/completions"
payload = {
    "model": "Qwen/Qwen3-4B",
    "messages": [
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "請解釋量子力學的基本概念"}
    ],
    "stream": True,
    "temperature": 0.7,
    "max_tokens": 1024,
    "session_id": "user-123",          # 啟用對話記憶
    "enable_thinking": False           # 關閉 thinking 模式
}

with httpx.Client(timeout=300) as client:
    with client.stream("POST", url, json=payload) as resp:
        for line in resp.iter_lines():
            if line.startswith("data: ") and line != "data: [DONE]":
                chunk = json.loads(line[6:])
                delta = chunk["choices"][0]["delta"].get("content", "")
                print(delta, end="", flush=True)

Tool Call 範例

# Tool Call (Function Calling)
payload = {
    "messages": [{"role": "user", "content": "今天台北天氣如何？"}],
    "tools": [
        {
            "type": "function",
            "function": {
                "name": "get_weather",
                "description": "取得指定城市的天氣",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "city": {"type": "string", "description": "城市名稱"}
                    },
                    "required": ["city"]
                }
            }
        }
    ],
    "tool_choice": "auto"
}

4. 程式碼結構詳解

理解專案結構有助於快速定位問題。

檔案路徑 角色與功能
service/app.py 入口點。定義所有 API 路由、CORS、生命週期管理、OpenAI 相容層（session 組合、串流處理）。
service/model_manager.py 中介者。Singleton，協調 API 層與 Worker Process 通訊。提供 start_loading, generate_stream, generate, stop_generation 等方法。
service/inference/model_inference_process.py Process 控制器。管理 multiprocessing.Process，控制 Worker 生命週期及 IPC Queue 通訊。
service/inference/engines/ 引擎實作。
  • base_engine.py: 抽象基類
  • transformers_engine.py: HF Transformers 引擎
  • vllm_engine.py: vLLM 引擎（子程序管理）
  • llama_server_engine.py: llama-server 引擎
service/inference/generate/generator_worker.py 生成邏輯。包含 handle_generate_request 和 handle_generate_stream_request，調用 model.generate() 並處理 Streamer 輸出。
service/inference/load_model/peft_loader.py PEFT 加載器。支援加載微調後的 LoRA/QLoRA adapter 模型（合併至基礎模型或以 PeftModel 形式加載）。
service/config_models.py 資料模型。定義所有 Pydantic 請求/回應 Schema，包含 InferenceConfig, OpenAIChatCompletionRequest, TrainingConfig 等。
service/session_manager.py 會話管理。管理多輪對話歷史記錄，支援 Redis 儲存（有 Redis 時）或記憶體儲存（fallback）。
service/model_registry.py 模型登錄。管理可用模型清單（base/finetuned/llama_gguf），儲存於 service/configs/models_registry.json。
service/download_manager.py 下載管理。非同步從 HuggingFace Hub 下載模型並自動註冊至 Model Registry。
service/utils/conversion_manager.py 格式轉換。將 HF 模型轉換為 GGUF 格式（呼叫 llama.cpp 轉換腳本）。
service/inference/memory_estimator.py 記憶體估算。根據模型大小與量化設定，估算各 Offload 策略下的 VRAM 需求。
service/settings.py 環境配置。讀取 .env 中的 HF_HOME、Redis、服務 Port 等設定。

5. 推理流程細節 (Deep Dive)

Step 1: 模型加載流程 (Load Flow)

  1. Client 發送 POST /inference/load_model（帶 engine, model_name, quantization 等）。
  2. app.py 檢查是否已有相同配置的模型在運行（若是則 already_loaded），或者是否有不同的模型（則 HTTP 409 要求先卸載）。
  3. app.py 呼叫 model_manager.start_loading(config)。
  4. ModelManager 將 load 指令推送到 Request Queue（IPC）。
  5. Worker Process 根據 engine 選擇對應引擎：
    • Transformers: 使用 BitsAndBytesConfig、AutoModelForCausalLM、Device Map 載入。
    • vLLM: 啟動 vllm serve 子程序，等待健康檢查通過。
    • llama-server: 啟動 llama-server 子程序，等待 /health 就緒。
  6. Worker 持續透過 Status Queue 回報進度（loading → ready）。

Step 2: OpenAI 相容對話流程 (Chat Flow)

注意： /v1/chat/completions 在主進程中負責 Session 管理和 Prompt 組裝，推理本身在 Worker Process 執行。
  1. Client 發送 POST /v1/chat/completions（帶 messages, stream=True, session_id）。
  2. app.py 執行 Prompt 組裝：
    • 若有 session_id，透過 session_manager 取得歷史（最多 6 則訊息，約 3 輪對話）。
    • 解析 enable_thinking 並套用 Model-aware 生成參數（如 Qwen3.5 自動調整 temperature/top_p/repetition_penalty）。
  3. 若請求包含 tools 或 tool_calls，走 Tool Passthrough 路徑（繞過 session 管理，直接透傳）。
  4. app.py 呼叫 model_manager.generate_stream()，取得迭代器。
  5. Worker Process 執行生成，使用 TextIteratorStreamer 逐 Token 推送到 Data Queue。
  6. app.py 透過 SSE 將 Data Queue 內容即時包裝成 OpenAI chunk 格式回傳給前端。
  7. 生成完成後，app.py 將本輪 user/assistant 對話追加到 session_manager。

Qwen3.5 Model-Aware 參數自動調整

特殊行為： 當偵測到載入的模型為 Qwen3.5 系列（model name 含 "qwen3.5"）時，系統會自動強制覆蓋生成參數：
  • temperature 強制設為 1.0
  • top_p 強制設為 0.95
  • top_k 強制設為 20
  • repetition_penalty 強制設為 1.0
  • enable_thinking 強制設為 false
這是為了避免 Qwen3.5 MoE 架構在非官方參數下產生多語言亂碼問題。

7. 模型管理 (Model Registry)

模型清單存於 service/configs/models_registry.json，分三類型：base（HF 模型）、finetuned（微調後）、llama_gguf（GGUF）。

查詢可用模型

GET /config/models

返回所有已登錄的基礎模型、微調模型、GGUF 模型清單。

下載模型

POST /config/models/download

從 HuggingFace Hub 下載模型並自動登錄至 registry。非同步執行，返回 task_id，可透過 GET /config/models/download/{task_id} 查詢進度。

{
  "model_id": "Qwen/Qwen3-4B",
  "label": "qwen3_4b",
  "force_download": false
}

刪除模型

DELETE /config/models/{label}

從 registry 移除模型。加上 ?delete_files=true 可同時刪除本地檔案（含 HF cache 整個目錄）。

轉換 GGUF

POST /config/models/convert

將 HF 模型轉換為 GGUF 格式，非同步執行。支援 outtype: auto/f16/bf16/q8_0/f32。

8. 常見問題與處理

❌ Error: Out of Memory (OOM)

症狀: Worker Process 崩潰，/inference/status 顯示 loading_error。

解決方案:

  • 呼叫 POST /inference/force_cleanup_gpu 強制清理 VRAM。
  • 降低 max_memory GPU 限制，強制更多層 Offload 到 CPU。
  • 改用更高的量化等級 (int8 → int4)。
  • 改用 vllm_cpu_offload_gb（vLLM 引擎）。
  • 先呼叫 /inference/estimate_memory 評估需求再決定配置。

❌ HTTP 409: Model Already Loaded

症狀: 嘗試載入不同模型時收到 409。

解決方案: 先呼叫 POST /inference/unload_model，再重新加載。

❌ HTTP 410: Deprecated Endpoint

症狀: 呼叫 POST /inference/chat 收到 410。

解決方案: 改用 POST /v1/chat/completions，格式相容 OpenAI API。

❌ Inference Timeout

症狀: 生成時間過長導致連線中斷。

解決方案:

  • 調低 max_tokens（對應 max_new_tokens）參數。
  • 在請求中調高 total_timeout（預設 300 秒）。
  • 檢查是否過多層在 CPU Offload（CPU 推理極慢）。

❌ Tokenizer / Template Error

症狀: 回應內容包含奇怪的標籤或格式錯誤。

解決方案:

  • 確認 trust_remote_code: true 已啟用。
  • 檢查模型是否需要特殊的 chat template（vLLM 可透過 vllm_chat_template 指定自訂 .jinja 檔）。
  • 對於 Qwen3.5，系統已自動禁用 thinking mode，如仍有問題請確認 API 請求中未強制設定 enable_thinking: true。

❌ vLLM 啟動失敗 / Port Conflict

症狀: vLLM 引擎載入失敗，log 顯示 port 被佔用。

解決方案:

  • 服務啟動時會自動執行 sweep_stale_vllm_processes() 清理殘留進程。
  • 手動確認端口占用：lsof -i :5000（vLLM 預設埠 VLLM_PORT），必要時 kill -9 {PID}。
  • 重啟 FastAPI 服務後再嘗試載入。
